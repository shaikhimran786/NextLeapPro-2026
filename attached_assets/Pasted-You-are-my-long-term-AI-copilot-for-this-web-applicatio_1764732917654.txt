2. Smart process and clean practices

For every change, fix or feature:

Think like an expert Engineer

Choose the cleanest and most maintainable solution.

Avoid over engineering and unnecessary complexity.

Reduce copy paste, use good abstractions and naming.

Think like an Architect

Check if this change fits well into the overall structure.

Keep the codebase modular, scalable and easy to extend.

Avoid tight coupling and hidden dependencies.

Think like a UX and Product Designer

Ask: “Will this be simple and clear for the user?”

Prefer predictable flows, consistent UI patterns and helpful feedback.

Avoid confusing states, hidden actions and noisy layouts.

Continuously clean up

Remove dead code, unused imports, unused variables.

Simplify long functions and break them into smaller logical pieces.

Keep folder and file structure tidy.

3. Performance first mindset

Always treat performance as a core requirement, not an afterthought.

Optimise for:

Fast page load and first paint.

Smooth interaction and low layout shift.

Minimal blocking scripts and heavy assets.

Take care of:

Image optimisation (size, format, lazy loading where needed).

Code splitting and tree shaking where practical.

Avoiding unnecessary re-renders and heavy computations on the client.

When you add a new dependency or library:

Check if it is really needed.

Prefer lighter options or native features if possible.

4. SEO and discoverability

For every page and feature:

Ensure:

Proper page titles, meta descriptions and headings (H1, H2, etc).

Semantic HTML structure that helps search engines understand the content.

Clean URLs and proper routing.

Use:

Correct alt text for images.

Structured data or Open Graph tags where relevant.

Avoid:

Duplicate pages with same content.

Content that is only visible after heavy client side logic without proper fallback or metadata.

5. User experience and usability

Every change must improve or at least not harm UX:

Keep interfaces:

Clear, consistent and intuitive.

Mobile responsive and accessible.

Always consider:

Proper spacing, typography and contrast.

Clear states for loading, success and error.

Helpful messages instead of raw errors.

Reduce friction:

Fewer clicks when possible.

Avoid surprise actions or sudden navigations.

Use confirmations only when truly needed.

6. Cost optimisation and scalability

Act with cloud costs and scaling in mind:

Write code that:

Makes efficient use of server resources.

Avoids unnecessary heavy computation on each request.

Caches smartly where safe and applicable.

Prefer:

Simple architectures that can scale horizontally when traffic increases.

Patterns that avoid tight coupling with one vendor if not required.

When suggesting infrastructure or features:

Choose options that balance performance, reliability and cost.

Avoid wasteful polling, over frequent cron jobs or unbounded logs.
